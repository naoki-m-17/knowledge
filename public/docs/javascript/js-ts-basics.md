# JavaScript / TypeScript の共通ルールと文法

## 命名慣習（パスカルケース：大文字から始める）

型定義 (interface, type)：データの「設計図」であることを示す。

クラス (class)：「実体を作る元」であることを示す。

## 命名慣習（キャメルケース：小文字から始める）

変数・定数：データそのもの。

関数（処理）：ロジック（動き）をまとめたもの。

**注意**: React では、コンポーネント名は大文字で始めるという特別なルールがあります。これは JavaScript/TypeScript の一般的な命名規則とは異なる、React 独自のルールです。詳細については、[React の命名規則](../react/naming-conventions.md) を参照してください。

## export と定義を 1 行で書く

【区分：JS/TS の仕様】

理由：ES Modules の仕様。定義と同時に公開することで書き忘れを防ぐ。モダンな開発の標準マナー。

## export default と const が 1 行で書けない理由

【区分：JS/TS の文法制限】

言語そのものの制約。そのため「最後で export default する」か「default を使わず export const（名前付き）にする」かの 2 択になる。

## interface と type の違い・切り出し

【区分：TypeScript の特徴】

TS 独自の機能。大規模な拡張が必要なら interface、単純な型や組み合わせなら type など、コード整理術の観点で使い分ける。

## ディレクトリ構成と型の管理ルール

### type / types ディレクトリ

**【区分：言語（TypeScript）の運用】**

役割：純粋な型定義（interface / type）のみを置く。

特徴：ビルド（JavaScriptへの変換）時には完全に消滅し、実行時の動作には一切影響を与えない「純粋な設計図」。

### lib ディレクトリ

**【区分：共通ロジックの管理】**

役割：実体のある処理（計算、データ加工、ファイル操作など）を置く。

特徴：実際にブラウザやサーバーで動く「JavaScriptのコード（関数）」が含まれる。

### 使い分けの判断（モダンな設計思想）

**グローバルな型**: アプリ全体で使い回す基本構造（User型など）は `types/` で一括管理。

**ローカルな型（コロケーション）**: 特定の lib 関数やコンポーネントでしか使わない型は、そのファイル内に同居させる。

理由：関連するものが近くにある方が、コードを修正する時に迷わず、保守性が高まる（「凝集度」を高めるという考え方）。
