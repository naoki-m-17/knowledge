# リソース限界時の挙動：メモリ帯域幅とスワップ

メモリ帯域幅（データの通り道の太さ）の違いは、限界域での挙動に現れる。

## メモリ帯域幅の比較

| 項目 | **M4 Pro** | **M5** |
| --- | --- | --- |
| **メモリ帯域幅** | **273GB/s** | 153GB/s |
| **ユニファイドメモリ** | あり | あり |
| **帯域幅の実感** | 大容量データの瞬時移動 | 一般的な用途では十分 |

## 帯域幅が限界に達した時の挙動

帯域幅が飽和すると、データの読み書きに「待ち時間」が発生する。これはPCがフリーズするのではなく、高負荷なデータの入れ替えが「全体的にワンテンポ遅れる」粘りつくような感覚として現れる。

### M4 Pro (273GB/s) が真価を発揮する場面

- **4K動画の多重レイヤー編集**: 複数の4Kトラックを同時にプレビューする
- **高精細な3Dレンダリング**: 複雑なシーンをリアルタイムでレンダリングする
- **数千枚のRAW現像**: 大量の高解像度画像を連続処理する
- **大規模なデータセットの処理**: 機械学習の前処理、大量のCSVファイルの解析

これらの作業では、帯域幅の広さが直接的な処理時間の短縮に繋がる。

### M5 (153GB/s) で十分な場面

- **Webブラウジング**: 一般的なWebサイトの閲覧、SNSの利用
- **オフィスワーク**: Word、Excel、PowerPointでの作業
- **一般的な2Dデザイン**: Figma、Sketch、Adobe Illustratorでの作業
- **軽量なプログラミング**: テキストエディタでのコーディング、軽量なビルド

153GB/sという数値自体、かつてのハイエンド機を凌駕する水準にある。4K動画を数本同時にプレビューするような巨大データの転送をしない限り、M5の153GB/sで「道路が渋滞する」ことはまずない。

## メモリ不足による「スワップ」発生時

メモリ容量（32GB等）を超えた場合、SSDをメモリ代わりにする「スワップ」が発生する。この時の速度低下は帯域幅よりもSSDの転送速度に依存する。メモリ容量の重要性については、[エンジニア向け視点](./engineer-perspective.md)でも詳しく説明している。

### スワップが発生する場面

- **多数のアプリケーションを同時起動**: ブラウザタブを50以上開く、複数のIDEを起動する
- **大規模なプロジェクトのビルド**: メモリを大量に消費するコンパイル作業
- **仮想環境の同時稼働**: 複数のDockerコンテナやVMを起動する

### スワップ時の挙動

- **帯域幅の影響は限定的**: SSDの読み書き速度（約3,000MB/s〜7,000MB/s）がボトルネックとなる
- **体感**: アプリケーションの起動が遅くなる、操作にラグが生じる
- **対策**: メモリ容量を増やす（32GB以上）ことが最も効果的

**結論**: 帯域幅の違いは、メモリが十分にある場合の「処理速度」に影響するが、メモリ不足時の「スワップ速度」にはほとんど影響しない。メモリ容量の確保が優先される。コア数の余力による影響については、[マルチタスクとコア数の余力](./multitasking.md)を参照。
